敏捷开发模式下的Github流程管理方案 ver2.0

发表于 2018-01-18 | | 阅读次数:

去年底确定的第一版方案实行了一段时间后，发现大家在很多细节操作上，还存在疑问，因而，第二版对部分具体场景进行了进一步的细化，并添加了对Commit与Pull request描述的一些要求。

基本原则：Repository不做权限限制，部分分支开启分支保护

分支情况

（需要特别注意develop分支，与主流方式有较大差异）

  • 必需分支：
    • master分支：主分支，任何时候在这个分支拿到的，都是稳定的发布版，即线上版本（可能存在一些仅有master分支存在的代码，如Website GA等），保险起见，merge到该分支必要上线 （需要分支保护）
    • release分支：预发布分支，每个项目基于前一个上线release分支创建新release分支，长期保留分支 （需要分支保护）
    • develop分支：开发分支（由各开发人员自行创建），从前一个已上线release的分支，根据开发人员或功能点创建大量develop分支，在开发完成并进行code review后，merge至相应release分支并提测，后续的bug修复在该develop分支上进行，修复后merge至release分支，在项目完成后删除各分支
  • 非必需分支
    • hotfix分支：补丁分支，特殊形式的开发分支，只在线上需要修复问题时从release分支创建，在该分支修复线上问题，merge回master分支及前一个上线release分支后，可删除该分支（若该时间点下一个release分支已创建，则还需要merge至下一个release分支）
    • feature分支：功能分支，特殊形式的开发分支，只在开发长期且不确定项目时创建，merge回develop分支后，可删除该分支
  • 分支命名规则
    • master分支：master
    • release分支：release/v1.2.18
    • develop分支：dev/v1.2.18_xxxxxx
    • feature分支：feature/xxxxx/20180101

正常开发流程说明

  • 每个人在自己的develop分支上提交代码（多人合作功能同一个develop分支进行）
  • 开发完成，需要进行Code review时，创建New Pull Request，申请将开发完成的develop分支merge至对应release分支
    • Reviewer在review该代码后，若不存在问题，点击approve，Admin确认merge操作
    • Reviewer在review该代码后，若存在问题，则dismiss该pull request，由开发人员修改后，重新发起
    • 若pull request存在代码冲突，则需要开发人员先在自己的IDE上将release分支merge至自己的develop分支后，重新发起pull request，不再release分支上处理冲突
  • 将要部署本地测试环境时，部署release分支
  • 在部署本地测试环境后进行的问题修改，开发人员需要先将release分支merge至develop分支后，继而在develop分支上进行修改，release分支始终只接受merge方式提交代码
  • 在项目上线后，相应的develop分支群删除，后续bug新建hotfix修改并merge至develop、release分支

hotfix流程说明

  • 基于产线release分支创建hotfix分支
  • 在hotfix分支上进行修改
  • 在需要部署本地测试时，将hotfix分支merge至release分支进行部署
  • 在需要部署产线时，将release分支merge至master分支进行部署
  • 在hotfix通过release分支merge上线后，可删除相应hotfix分支

线上hotfix对后续项目Release分支的影响

  • 设定场景：线上对应分支为release/v1.2.16，在后续项目开发进行过程中，hotfix上线，后续项目对应分支为release/v1.2.17，在hotfix上线后，将release/v1.2.16向release/v1.2.17merge时，发现冲突，由于release/v1.2.16分支存在分支保护，无法在IDE进行merge操作，因而需要特殊处理
  • 解决冲突方案：
    • 基于release/v1.2.17创建临时分支forMerging/v1.2.17
    • 将release/v1.2.16分支merge至forMerging/v1.2.17，在IDE解决冲突并提交
    • 将forMerging/v1.2.17提Pull Request，Merge至release/v1.2.17，由于release/v1.2.16与release/v1.2.17的冲突已在forMerging/v1.2.17分支解决，此次merge必然不会出现冲突
    • 至此冲突问题已解决，若开发人员develop分支提Pull Request时出现冲突，则参照前文描述，先merge再Pull Request的方式处理，不会受release/v1.2.16与release/v1.2.17之间冲突的影响

Commits & Pull Request描述要求

  • Commits
    • 格式：Type(Scope) Target：Description
    • 可选Type类型（必填）：
      • feat：新功能（feature）
      • fix：修补bug
      • docs：文档（documentation）
      • style： 格式（不影响代码运行的变动）
      • refactor：重构（即不是新增功能，也不是修改bug的代码变动）
      • test：增加测试
      • chore：构建过程或辅助工具的变动
    • Scope（可选）:用于说明 commit 影响的范围，比如数据层、控制层、视图层等等，视项目不同而不同
    • Target（可选）：若为issue则为issue编号，若为
    • Description（必填）参考：add/update/change/delete... [xxxx] in [file] for [reason]
    • 示例：
      • 新功能的开发：feat #1245：[description]
      • 针对Issue的bug修改：fix #1246：[description]，若Issue标题能够说清问题，Description可以复制Issue的标题
      • 无对应Issue的bug修改：fix：[description]
      • Release note修改：docs：[description]
  • Pull Request
    • Subject格式：dev/v1.2.17 merge to release/v1.2.17 [Title of Feature]
    • Detail格式：参考Commit

优点

  • 可对每部分提交独立进行code review
  • 部分功能代码可offshore code review，通过github提供的功能进行review流程，在线即可推进，既避免等待也不会出现遗漏
  • 降低代码质量不稳定对项目造成的风险，保证release分支代码的可靠性
  • 减少merge时冲突的可能性
  • DM可以通过pull request掌握项目开发进度
  • 分阶段开发/多项目并行的情况下也可行

Q&A

  • Q：创建的分支太多，管理不便
    • A：由各开发人员自行创建，个人对自己的分支负责，DM只负责release分支及对release分支的pull request，除特殊情况业务交接，基本不会存在处理大量分支的情况
  • Q：develop开发阶段修改bug切换分支可能会很麻烦
    • A：通过提高质量、减少发版次数避免
  • Q：多个pull request是否存在先后顺序问题，若提交的pull request之间存在冲突如何处理
    • A：开发A与开发B同时对文件A进行修改，并提Pull Request，若Reviewer先对开发A的Pull Request进行Review，并通过完成Merge，则开发B的Pull Request会自动被检测出冲突，需要开发B将Release分支merge至自己的develop分支，即可解决冲突问题

注意

由于release分支存在分支保护，向release分支Pull request产生的冲突，无法在IDE上解决，因而需要开发人员将release分支merge至自己的develop分支后，重新pull request（在compare阶段发现冲突后，即可停止pull request）
