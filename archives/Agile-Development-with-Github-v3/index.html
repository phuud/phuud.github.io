敏捷开发模式下的Github流程管理方案 ver3.0

发表于 2018-05-31 | | 阅读次数:

基于一二版的方案实行近半年后，发现部分流程在实际操作中并没有得到落实，重新讨论后决定对流程进行一定程度的精简，去除不符合实际操作需要的步骤和要求，并添加了一些对Code Review的要求。

基本原则：保持分支职能划分，取消分支保护，简化部分流程

分支情况

（需要特别注意develop分支，与主流方式有较大差异）

  • 必需分支：
    • master分支：主分支，任何时候在这个分支拿到的，都是稳定的发布版，即线上版本（可能存在一些仅有master分支存在的代码，如Website GA等），保险起见，merge到该分支必要上线 （需要分支保护）
    • release分支（大版本）：预发布分支，每个项目启动时，由DM基于前一个上线release分支创建的新release分支，长期保留分支
    • develop分支：开发分支，在开发工作开始前，由各开发人员自行创建根据需求功能点，基于DM创建的当前项目release分支创建一个或数个develop分支，在开发完成并进行code review后，merge至相应release分支并提测，后续的bug修复可在该develop分支上进行，也可在DM创建的Bug集中修复分支（dev/v1.2.xx_fixbug）上进行，修复后merge至release分支，在项目完成后删除各分支
  • 非必需分支
    • release分支（小版本）：针对Hotfix的预发布分支，在确定需要上线Hotfix时，由DM基于前一个上线release分支创建的新release分支，长期保留分支
    • hotfix分支：补丁分支，特殊形式的开发分支，只在线上需要修复问题时从release分支（小版本）创建，在该分支修复线上问题，merge回master分支及相应release分支（小版本）后，可删除该分支（若该时间点下一个release分支已创建，则还需要merge至下一个release分支）
    • feature分支：功能分支，特殊形式的开发分支，只在开发长期且不确定项目时创建，merge回develop分支后，可删除该分支
  • 分支命名规则
    • master分支：master
    • release分支（大版本）：release/v1.2.18
    • develop分支：dev/v1.2.18_xxxxxx
    • release分支（小版本）：release/v1.2.18.x
    • feature分支：feature/xxxxx/20180101

正常开发流程说明

  • 每个人在自己的develop分支上提交代码（多人合作功能同一个develop分支进行）
  • 开发完成，需要进行Code review时，创建New Pull Request，申请将开发完成的develop分支merge至对应release分支
    • Reviewer在review该代码后，若不存在问题，点击approve，Admin确认merge操作
    • 若pull request存在代码冲突，则需要开发人员先在自己的IDE上将release分支merge至自己的develop分支后，重新发起pull request，不再release分支上处理冲突
  • 将要部署本地测试环境时，部署release分支
  • 在部署本地测试环境后进行的问题修改，开发人员需要使用fix bug专用的develop分支进行修改，release分支始终只接受merge方式提交代码
  • 在项目上线后，相应的develop分支群删除，后续bug新建hotfix修改并merge至develop、release分支

Code Review流程说明

开发人员在开发并自测完成后，创建New Pull Request，申请将开发完成的develop分支merge至对应release分支，存在以下3种Code Review形式：

  • 使用Github Pull Request功能进行Review，针对修改很少或影响面很小的问题
  • 一对一Review，针对新人导师及其他由开发人员判断需要的情况
  • Code Review Meeting，以下4种情况必须以会议形式进行Reveiw：
    • 全新业务/技术功能的实现
    • 初次代码重构
    • 核心业务的修改
    • 新员工前2~3次代码

hotfix流程说明

  • DM基于产线release分支创建新的release分支（小版本），开发人员基于release分支（小版本）创建hotfix分支
  • 在hotfix分支上进行修改
  • 在需要部署本地测试时，将hotfix分支merge至release分支（小版本）进行部署
  • 在需要部署产线时，将release（小版本）分支merge至master分支进行部署
  • 在hotfix通过release分支（小版本）merge上线后，可删除相应hotfix分支

线上hotfix对后续项目Release分支的影响

  • 设定场景：线上对应分支为release/v1.2.16，在后续项目开发进行过程中，hotfix上线，后续项目对应分支为release/v1.2.17，在hotfix上线后，将release/v1.2.16向release/v1.2.17merge时，发现冲突，由于release/v1.2.16分支存在分支保护，无法在IDE进行merge操作，因而需要特殊处理
  • 解决冲突方案：
    • 基于release/v1.2.17创建临时分支forMerging/v1.2.17
    • 将release/v1.2.16分支merge至forMerging/v1.2.17，在IDE解决冲突并提交
    • 将forMerging/v1.2.17提Pull Request，Merge至release/v1.2.17，由于release/v1.2.16与release/v1.2.17的冲突已在forMerging/v1.2.17分支解决，此次merge必然不会出现冲突
    • 至此冲突问题已解决，若开发人员develop分支提Pull Request时出现冲突，则参照前文描述，先merge再Pull Request的方式处理，不会受release/v1.2.16与release/v1.2.17之间冲突的影响

Commits & Pull Request描述要求

  • Commits
    • 格式：Type Target：Description
    • 可选Type类型（必填）：
      • feat：新功能（feature）
      • fix：修补bug
    • Target（若存在则为必填）：若为JIRA ticket，则为NTE-xx编号
    • Description（可选）：修改描述，尽量填写
  • Pull Request
    • Subject格式：[Current Version]dev/v1.2.17 merge to release/v1.2.17
    • 说明：
      • Current Version：当前版本号，如’v1.2.17’
    • Detail格式（可选）：Pull request描述

Q&A

  • Q：创建的分支太多，管理不便
    • A：由各开发人员自行创建，个人对自己的分支负责，DM只负责release分支及对release分支的pull request，除特殊情况业务交接，基本不会存在处理大量分支的情况
  • Q：develop开发阶段修改bug切换分支可能会很麻烦
    • A：通过提高质量、减少发版次数避免
  • Q：多个pull request是否存在先后顺序问题，若提交的pull request之间存在冲突如何处理
    • A：开发A与开发B同时对文件A进行修改，并提Pull Request，若Reviewer先对开发A的Pull Request进行Review，并通过完成Merge，则开发B的Pull Request会自动被检测出冲突，需要开发B将Release分支merge至自己的develop分支，即可解决冲突问题

注意

由于release分支存在分支保护，向release分支Pull request产生的冲突，无法在IDE上解决，因而需要开发人员将release分支merge至自己的develop分支后，重新pull request（在compare阶段发现冲突后，即可停止pull request）
