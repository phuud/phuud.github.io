敏捷开发模式下的Github流程管理方案 ver4.0

发表于 2018-09-05 | | 阅读次数:

由于近期Scrum Team结构和业务线的调整，使得工作流程出现了较大的变化，因而之前实行的方案已无法继续适用，讨论后做出了一定调整，并对开发流程进行了较细致的描述。

基本原则：保持分支职能划分，简化部分流程，添加删除分支步骤，责任到人

分支情况

  • 必需分支：
    • master分支：主分支，任何时候在这个分支拿到的，都是稳定的发布版，即线上版本（可能存在一些仅有master分支存在的代码，如Website GA等） （需要分支保护）
    • release分支（大版本/小版本流程相同）：预发布分支，每个项目启动时，由DM基于前一个上线release分支/Tag创建的新release分支，在上线、Merge至Master分支并创建Tag后，删除
    • release_baseline分支：在集成测试前，各team使用的预发布分支，每个项目启动时，由DM基于当前release分支创建新的baseline分支，在Merge至相应release分支后可删除
    • develop分支：开发分支，在开发工作开始前，由各开发人员自行创建根据需求功能点，基于DM创建的当前项目release_baseline分支创建一个或数个develop分支，在开发完成并进行code review后，merge至相应release_baseline分支后删除；后续的bug修复工作可基于release_baseline分支创建新的develop分支进行，也可在DM创建的Bug集中修复分支上进行，修复后merge至release_baseline分支，并删除该分支
  • 非必需分支
    • hotfix分支：特殊形式的develop分支，当确定要通过Hotfix修复线上问题，但又无法确定上线使用的具体小版本号时，需要从release分支创建hotfix，并在该分支上修复线上问题，在确定上线版本号时，merge回相应的release分支（小版本）后，即可删除该分支
    • feature分支：功能分支，特殊形式的开发分支，只在开发长期且不确定项目时创建，merge回develop分支后，可删除该分支
  • 分支命名规则
    • master分支：master
    • release分支（大版本）：release/v1.2.18
    • release分支（小版本）：release/v1.2.18.x
    • release_baseline分支（小版本）：release/v1.2.18baseline{team_name}
    • develop分支：dev/v1.2.18_xxxxxx
    • hotfix分支：hotfix/v1.2.18_{feature_name}
    • feature分支：feature/xxxxx/20180101

正常开发流程说明

  • 项目开始
    • 各项目Owner根据当前项目的改动情况，基于上一个项目Release的Tag创建新的release分支，并Slack通知
    • 各Team的DM基于当前release分支，创建自己Team使用的对应版本release_baseline分支，并Slack通知
    • （部分不常改动的Service，需要开发人员主动联系相应项目Owner，并阐明自己的需求及修改方案，要求项目Owner完成分支创建工作）
  • 开发过程中
    • 每个人基于当前的release_baseline分支，创建自己的develop分支，并在自己的develop分支上提交代码（多人合作功能同一个develop分支进行）
    • 完成某个功能，需要进行Code review时，创建Pull Request，申请将开发完成的develop分支merge至对应release_baseline分支
      • Reviewer在review该代码后，若不存在问题，点击approve，Admin确认merge操作
      • 若Pull request存在代码冲突，则需要开发人员先在自己的IDE上将release_baseline分支merge至自己的develop分支后，重新发起pull request，尽量避免在release_baseline分支上处理冲突
      • Reviewer在完成Merge工作后，会直接删除改dev分支，并通知相应开发使用新的分支
  • 部署测试
    • 各Team在部署本地测试环境时，部署自己的release_baseline分支
    • 在完成本地测试后，将各Team的release_baseline分支merge至当前项目的release分支，并在集成测试（Test）环境部署release分支
    • 在部署本地/集成测试环境后进行的问题修改，开发人员需要创建新的分支或使用fix bug专用的develop分支进行修改，release/release_baseline分支始终只接受merge方式提交代码
  • 上线部署前，DM通过Slack通知各项目Owner即将上线，需要检查：
    • 当前版本各项目的release_baseline是否已merge至release
    • 各项目的最新release分支是否已包含所有代码（有必要的需要将master分支merge回release分支）
    • 完成当前版本的配置文件对比工作，
  • 上线并完成测试后
    • DM发送项目产线上线邮件
    • DM通过Slack通知各项目Owner完成各项目后续工作，包括：release分支merge至master、基于当前release分支创建Tag、删除相应的release分支，完成后Slack回复
    • DM在确认各项目Owner完成工作后，更新Confluence信息页面，并发送邮件抄送所有开发，附带更新后的Confluence页面链接
    • 各开发人员删除自己在当前项目中创建的已失效develop/feature分支

Code Review流程说明

开发人员在开发并自测完成后，创建New Pull Request，申请将开发完成的develop分支merge至对应release分支，存在以下3种Code Review形式：

  • 使用Github Pull Request功能进行Review，针对修改很少或影响面很小的问题
  • 一对一Review，针对新人导师及其他由开发人员判断需要的情况
  • Code Review Meeting，以下4种情况必须以会议形式进行Reveiw：
    • 全新业务/技术功能的实现
    • 初次代码重构
    • 核心业务的修改
    • 新员工前2~3次代码

线上hotfix对后续项目Release分支的影响

  • 设定场景：线上对应分支为release/v1.2.16，在后续项目开发进行过程中，hotfix上线，后续项目对应分支为release/v1.2.17，在hotfix上线后，将release/v1.2.16向release/v1.2.17merge时，发现冲突，由于release/v1.2.16分支存在分支保护，无法在IDE进行merge操作，因而需要特殊处理
  • 解决冲突方案：
    • 基于release/v1.2.17创建临时分支forMerging/v1.2.17
    • 将release/v1.2.16分支merge至forMerging/v1.2.17，在IDE解决冲突并提交
    • 将forMerging/v1.2.17提Pull Request，Merge至release/v1.2.17，由于release/v1.2.16与release/v1.2.17的冲突已在forMerging/v1.2.17分支解决，此次merge必然不会出现冲突
    • 至此冲突问题已解决，若开发人员develop分支提Pull Request时出现冲突，则参照前文描述，先merge再Pull Request的方式处理，不会受release/v1.2.16与release/v1.2.17之间冲突的影响

Commits & Pull Request描述要求

  • Commits
    • 格式：Type Target：Description
    • 可选Type类型（必填）：
      • feat：新功能（feature）
      • fix：修补bug
    • Target（若存在则为必填）：若为JIRA ticket，则为Ticket编号
    • Description（可选）：修改描述，尽量填写
  • Pull Request
    • Subject格式：[Current Version]dev/v1.2.17 merge to release/v1.2.17
    • 说明：
      • Current Version：当前版本号，如’v1.2.17’
    • Detail格式（可选）：Pull request描述

Q&A

  • Q：创建的分支太多，管理不便
    • A：由各开发人员自行创建，个人对自己的分支负责，DM只负责release分支及对release分支的pull request，除特殊情况业务交接，基本不会存在处理大量分支的情况
  • Q：develop开发阶段修改bug切换分支可能会很麻烦
    • A：通过提高质量、减少发版次数避免
  • Q：多个pull request是否存在先后顺序问题，若提交的pull request之间存在冲突如何处理
    • A：开发A与开发B同时对文件A进行修改，并提Pull Request，若Reviewer先对开发A的Pull Request进行Review，并通过完成Merge，则开发B的Pull Request会自动被检测出冲突，需要开发B将Release分支merge至自己的develop分支，即可解决冲突问题

注意

  • 由于release分支存在分支保护，向release分支Pull request产生的冲突，无法在IDE上解决，因而需要开发人员将release分支merge至自己的develop分支后，重新pull request（在compare阶段发现冲突后，即可停止pull request）
