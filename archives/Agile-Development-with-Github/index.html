敏捷开发模式下的Github流程管理方案

发表于 2017-12-30 | | 阅读次数:

近期根据公司项目流程的一些具体情况，对公司目前的Github管理方案进行了整理细化，确定了部分细节并添加了具体的规则。虽然还存在一些问题，但决定先试水执行一段时间看看效果。

基本原则：Repository不做权限限制，部分分支开启分支保护

分支情况

（需要特别注意develop分支，与主流方式有较大差异）

  • 必需分支：
    • master分支：主分支，任何时候在这个分支拿到的，都是稳定的发布版，即线上版本，保险起见，merge到该分支必要上线 （需要分支保护）
    • release分支：预发布分支，每个项目基于前一个上线release分支（或master分支）创建新release分支，长期保留分支 （需要分支保护）
    • develop分支：开发分支（由各开发人员自行创建），从前一个已上线release的分支（或master分支），根据开发人员或功能点创建大量develop分支，在开发完成并进行code review后，merge至相应release分支并提测，后续的bug修复在该develop分支上进行，修复后merge至release分支，在项目完成后删除各分支
  • 非必需分支
    • hotfix分支：补丁分支，特殊形式的开发分支，只在线上需要修复问题时从master分支创建，在该分支修复线上问题，merge回master分支后，可删除该分支
    • feature分支：功能分支，特殊形式的开发分支，只在开发长期且不确定项目时创建，merge回develop分支或release分支后，可删除该分支

正常开发流程说明

  • 每个人在自己的develop分支上提交代码（多人合作功能同一个develop分支进行）
  • Code review后，将该develop分支merge至release分支
  • 将要部署本地测试环境时，部署release分支
  • 在部署本地测试环境后进行的问题修改，一律在develop分支上进行，release分支始终只接受merge方式提交代码
  • 在项目上线后，相应的develop分支群删除，后续bug新建hotfix修改并merge产线

hotfix流程说明

  • 基于产线release分支（或master）创建hotfix分支
  • 在hotfix分支上进行修改
  • 在需要部署本地测试时，将hotfix分支merge至release分支进行部署
  • 在需要部署产线时，将hotfix分支merge至master分支进行部署
  • 在hotfix上线后，可删除相应hotfix分支

优点

  • 可对每部分提交独立进行code review
  • 部分功能代码可offshore code review，通过github提供的功能进行review流程，在线即可推进，既避免等待也不会出现遗漏
  • 降低代码质量不稳定对项目造成的风险，保证release分支代码的可靠性
  • 减少merge时冲突的可能性
  • DM可以通过pull request掌握项目开发进度
  • 分阶段开发/多项目并行的情况下也可行

存在问题

  • Q: 创建的分支太多，管理不便
    • A: 由各开发人员自行创建，个人对自己的分支负责，DM只负责release分支及对release分支的pull request，除特殊情况业务交接，基本不会存在处理大量分支的情况
  • Q: develop开发阶段修改bug切换分支可能会很麻烦
    • A: 通过提高质量、减少发版次数避免
