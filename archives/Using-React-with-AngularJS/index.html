在AngularJS项目中集成React

发表于 2018-11-23 | | 阅读次数:

公司有个老项目，自搭建以来，始终使用AngularJS 1.3.11版本，由于错过了最佳的升级时机，且长时间的需求叠加使得目前系统业务已经非常复杂，尽管框架版本已经非常陈旧，在判断无法简单可靠实现平滑升级的情况下，我们短时间内没有进行框架升级的计划。

另一方面，考虑到公司技术栈已经转向React，且新的React站点已经搭建，之后针对这个老项目提出的业务多为功能补充和完善，与其耗费较大的精力对陈旧的框架进行升级，我们更倾向于将React项目集成到当前的AngularJS项目中，以期可以让开发人员尽早投入React的开发工作中。

项目当前情况

  • 框架：AngularJS 1.3.11
  • 包管理器：Bower
  • 构建工具：Grunt
    • 主要插件：babel、uglify、watch、connect等

前期调研

简单检索确认两个方案：

  • ngReact
    • 优势：Star多，接入和使用方式简单，能查询到的相关文档相对较多
    • 劣势：Demo只提供了ES5的写法，且项目创建时间相对较早，基本不会再更新维护了
  • react2angular
    • 优势：ngReact文档也推荐使用，实现方式可以使React Component独立性更高
    • 劣势：需要使用AngularJS Component，即需要将AngularJS版本升级到1.5+

结合需求综合考虑，只要能够实现ngReact使用JSX和ES6的方案，我们更倾向于选择对目前项目影响相对较小的ngReact

实现方案

引入ngReact

由于目前项目使用Bower作为包管理器，因而继续选择使用bower install ngReact，同时会安装并引入react及react-dom（这里有坑，后文会说明）

安装其他所需插件

涉及到的新增插件有（使用npm安装）：

  • “antd”: “^3.10.7”,
  • “babel-plugin-css-modules-transform”: “^1.6.1”,
  • “babel-plugin-import”: “^1.11.0”,
  • “babel-preset-es2015”: “^6.24.1”,
  • “babel-preset-react”: “^6.24.1”,
  • “babel-preset-stage-0”: “^6.24.1”,
  • “babelify”: “^8.0.0”,
  • “grunt-browserify”: “^5.3.0”,
  • “react”: “^16.1.0”,
  • “react-dom”: “^16.1.0”,

配置Gruntfile.js

项目一直使用Babel进行ES6的转换，但之前并不需要考虑import/export语法和开发阶段的预编译，若要书写React Component的写法，就必须考虑这些问题，因而选择使用grunt-browserify+babelify的方案

总结

目前已知存在以下风险或问题

  • React的引用
    • 由于项目一直以来使用bower作为包管理器，因而在使用bower安装ngReact的同时，也会使用bower安装所需的react，但bower-react仓库已停止维护，版本停留在16.1.0
    • React Component中使用import方式声明React实际上会再次引入通过npm安装在node_modules文件夹下的另一套React，因而使得项目中同时引入了两套不同来源的React
    • 同时引入两套React，不仅使得打包文件变大，同时还导致了兼容问题，为配合使用bower安装的react，npm安装的版本必须配合降级到16.1.0

      这个问题，需要对目前的引用方式进行调整，实现直接对node_modules文件夹路径的直接引用才能解决，由于一些项目遗留问题，必须要对打包的配置再进行一些调整才能实现，目前暂时搁置

  • Ant Design 样式的引用
    • 目前使用 babel-plugin-import 时并没有引入样式，而是通过index.html文件直接标签引入的形式，实现方式还是过于粗糙

      并没有找到合适的方式，可能需要引入其他babel的plugin或定制化处理

  • 开发过程中的实时编译
    • 由于目前在开发阶段，也需要对代码进行预编译，因而在每次修改React代码时，都会触发watch中的指定的browserify:dist任务，相比于之前纯AngularJS项目开发时不需要等待的方式，略影响效率

      需要进一步优化触发的时机判断，并精简任务执行的步骤

目前的实现方面虽然能解决当前的使用问题，但由于Demo并没有涉及较多复杂操作，潜在还存在很多不安定的因素，需要通过项目中实际使用时的反馈来做进一步的处理。
